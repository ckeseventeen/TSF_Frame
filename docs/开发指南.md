## 1. 设计目标与四条硬性原则

TSF_Frame 是通用时序预测框架,当前业务层聚焦住房公积金(HPF)月度指标预测。所有通用能力(模型 / 特征 / 监控 / 可视化)都与业务**解耦**,可移植到工业、零售、能源等领域。

四条硬性原则:

1. **业务与框架隔离** — `tsf_frame.*`(通用) 与 `configs.*` + `business.*`(业务特例) 分离。换行业 = 新加一个 `BusinessAdapter` + 一份 `Config` dataclass,**不改框架代码**。
2. **src-layout 强制隔离** — 源码住在 `src/tsf_frame/`,测试不会意外读到源目录残余 import; `pip install -e .` 是唯一安装路径。
3. **配置即代码** — 所有超参/阈值用 `@dataclass` 描述, `to_adapter_config()` 转 dict 给运行时。可读 + 可 diff + IDE 补全。
4. **有监控才算闭环** — 不只出预测数,还要回答"模型今天是否健康"。`ModelMonitor` 把性能/漂移/规则/告警/持久化串成完整链路。

### 为什么 `configs/` 不在 `src/tsf_frame/` 里?

`configs/` 描述"某次跑这个框架要用的参数",不是"框架本身的能力"。放进 `tsf_frame` 会让"换业务"变成"改框架内部"。独立顶层包的好处:

- 新业务只需在 `configs/` 下加子包(例如 `configs/stock/`),不污染 `tsf_frame`
- `setup.py` 用混合 `package_dir` 把两个顶层包一起发布
- 导入侧干净: `from configs.hpf import HPFConfig` / `from tsf_frame.business import HPFAdapter`

## 3. 核心抽象

框架用 4 类抽象基类把所有可变点收敛了。

### 3.1 `BaseModel` (`tsf_frame.models.base_model`)

```python
class BaseModel(ABC, nn.Module):
    def __init__(self, config: Dict[str, Any]): ...
    @abstractmethod
    def fit(self, train_data, val_data=None, **kw) -> Dict[str, Any]: ...
    @abstractmethod
    def predict(self, test_data, **kw) -> np.ndarray: ...

    def predict_probabilistic(self, test_data, **kw) -> ProbabilisticPrediction: ...
    def save_model(self, path: str): ...
    def load_model(self, path: str): ...
    def _fit_residuals(self, y_true, y_pred): ...           # 残差缓存 (多输出按列)
    def _get_residual_interval(self, y_pred): ...           # 按列分位数构造区间
```

**`ProbabilisticPrediction(mean, lower, upper, std, samples)`** 是统一数据载体 — 无论残差/MC Dropout/分位数,监控/可视化/业务层都只看这个对象。

实现映射:
- `models/classical/ml_models.py` — 11 个 sklearn/XGB/LGB/CatBoost 模型
- `models/transformer/transformer_models.py` — LSTM / Transformer / Autoformer / iTransformer / TimesNet / DLinear (6 个)
- `models/moirai/` — 零样本基础模型(Salesforce Moirai)

### 3.2 `BaseBusinessAdapter` (`tsf_frame.business.base_adapter`)

```python
class BaseBusinessAdapter(ABC):
    @abstractmethod
    def preprocess(self, data, **kw) -> Tuple[pd.DataFrame, Dict]: ...
    @abstractmethod
    def postprocess(self, predictions, metadata, **kw) -> pd.DataFrame: ...
    @abstractmethod
    def validate_data(self, data) -> Tuple[bool, str]: ...
    @abstractmethod
    def get_business_metrics(self, y_true, y_pred) -> Dict[str, float]: ...
```

"业务"这一层是**防腐层**:
- 业务规则(非负/单调/缺失上限)集中在这里入口校验
- 归一化/反归一化细节(HPF 推荐 zscore,股票可能 minmax)不泄漏到模型
- 业务指标(HPF 的 YoY / QoQ / direction_accuracy)不污染通用 `MetricsCalculator`

### 3.3 `BaseMonitor` (`tsf_frame.monitoring.base_monitor`)

```python
class BaseMonitor(ABC):
    @abstractmethod
    def record(self, event: Dict[str, Any]) -> None: ...
    @abstractmethod
    def check_status(self) -> MonitoringStatus: ...
    @abstractmethod
    def reset(self) -> None: ...
```

通用 `ModelMonitor` 组合了 `PerformanceMonitor + 3 种 DriftDetector + DataQualityMonitor + RuleEngine + RetrainingTrigger + AlertManager + MetricStore`。HPF 业务规则通过 `@register_rule('R_HPF_*')` 装饰器注入,**不污染通用 monitoring 包**。

### 3.4 监控插件契约(6 个 ABC + 7 个注册表)

详见 `tsf_frame/monitoring/interfaces.py`:

| ABC | 注册装饰器 | 工厂 | 实现例 |
|------|-----------|------|--------|
| `MetricStore` | `@register_store` | `create_store` | InMemory / SQLite / Jsonl |
| `AlertChannel` | `@register_alert_channel` | `create_alert_channel` | Console / Logging / File / Callback / Store |
| `DriftDetector` | `@register_drift_detector` | `create_drift_detector` | Data / Concept / Prediction |
| `QualityChecker` | `@register_quality_checker` | `create_quality_checker` | Missing / Outlier / Schema / Frequency / Range |
| `RuleChecker` | `@register_rule` (函数) | (走 RuleEngine) | R1/R2/R3/R4 + 用户自定义 |
| `ReportGenerator` | `@register_report` | `create_report` | Text / Plot |
| (指标函数) | `@register_metric` | `get_metric_fn` | mae/mse/rmse/mape/picp/miw/winkler |

**新加任何插件:** 继承 ABC → 装饰器注册 → 任何位置用工厂取实例。详见 §11。

## 4. 端到端数据流

以 HPF 月度缴存预测为例(`pipelines/run_hpf_forecast.py`):

```
原始 DataFrame (date, monthly_deposit, monthly_withdrawal, loan_balance, ...)
        │
        ▼ HPFAdapter.validate_data()           业务规则 + schema 检查
        │
        ▼ HPFAdapter.preprocess()              异常值处理 + zscore 归一化 → metadata
        │
        ▼ create_feature_engineer(...)         Time + Lag + Rolling + Difference
        │                                       全部 fit→transform, 严格因果
        │
        ▼ train/val/test 时间顺序切分           不 shuffle, 不随机 CV
        │
        ▼ get_ml_model('xgboost', cfg)         BaseModel 子类
        │     .fit(train_data=(X_tr, y_tr), val_data=(X_v, y_v))
        │     .predict_probabilistic(X_te)     → ProbabilisticPrediction
        │
        ▼ HPFAdapter.postprocess(pred, meta)   反归一化 + 裁剪非负 + 政策情景
        │
        ▼ MetricsCalculator.calculate_all      MAE/RMSE/MAPE/R²
        │  adapter.get_business_metrics        direction/YoY/QoQ
        │
        ▼ ModelMonitor.record_prediction(...)  写 SQLite + perf 窗口 + drift buffer
        │  (异步真值到达后) settle_actual(...)  按 target_ts 回填 + 残差喂 concept_drift
        │  ModelMonitor.check_status()         触发告警 + 必要时生成报表
        │
        ▼ PredictionPlotter.{forecast,metrics,training_curves}_fig(save_path=...)
        │
输出: logs/outputs/hpf/*.png + logs/monitor/hpf_monitor.db + logs/runs/*.log
```

## 5. 监控子系统架构

```
┌────────────────────── ModelMonitor (协调器) ───────────────────────┐
│                                                                      │
│  record_prediction(timestamp, prediction, actual=None, ...):         │
│    ├─► store.insert_prediction()           持久化                    │
│    ├─► perf.update(target_ts → _Record)   OrderedDict 按 target_ts  │
│    ├─► data_drift.update(features)         (仅 update_drift=True)    │
│    ├─► prediction_drift.update(y_pred)     (同上)                   │
│    └─► concept_drift.update(residual)      (仅 actual ≠ None)       │
│                                                                      │
│  settle_actual(target_ts, y_actual):       异步真值到达             │
│    ├─► perf.fill_actual(target_ts, ...)    按时间戳精确定位         │
│    ├─► store.update_actual(...)            持久化回填                │
│    └─► concept_drift.update(residual)      残差喂入                 │
│                                                                      │
│  check_status() -> MonitoringStatus:                                 │
│    ├─► perf.current()                      最近 metric_window 条     │
│    ├─► data/concept/prediction_drift.detect()                        │
│    ├─► rule_engine.check()                 内置 + @register_rule    │
│    ├─► retraining_trigger.check()          5 默认规则 + 冷却        │
│    ├─► store.insert_metrics_snapshot()     落库                     │
│    └─► alert_manager.emit()                fan-out 到所有通道       │
│            ├─► ConsoleChannel  → stderr                              │
│            ├─► LoggingChannel  → stdlib logging                      │
│            ├─► FileChannel     → alerts.log (WARNING+)               │
│            ├─► CallbackChannel → 用户函数 (钉钉/邮件/...)            │
│            └─► StoreChannel    → 另写一个 store                     │
└──────────────────────────────────────────────────────────────────────┘
```

### SQLite Schema (`monitoring/stores.py`)

```sql
predictions      (id, model_id, ts, target, y_pred, y_lower, y_upper, y_actual)
metrics_snapshot (id, model_id, ts, name, value, window_size)
alerts           (alert_id, model_id, ts, level, source, message, details)
```

`journal_mode=WAL` 支持并发读写。短连接模式避免跨线程锁。

### 关键设计:`target_ts` 对齐 + 异步真值回填

HPF 月度跑批场景下,**预测和真值不同时到达**。`PerformanceMonitor` 用 `target_ts` 作为 `OrderedDict` 的键(不依赖位置索引),即使窗口满 + 真值乱序到达,**回填永远精确**。

```
T₀ 跑批  → record_prediction(target_ts=T₁, prediction=p1, actual=None)
T₁ 跑批  → record_prediction(target_ts=T₂, prediction=p2, actual=None)
                                ...
T₁ 真值到 → settle_actual(target_ts=T₁, y_actual=a1)
```

详见使用指南 §10.

### 冷启动降级

`_n_records < cold_start_samples` 时,只跑性能监控 + 规则,**跳过漂移检测**(样本不足会误报 PSI/error_trend)。

### 多目标 / 多步监控

- `MultiHorizonMonitor`: 单 target × 12 horizon, 每个 horizon 独立 `PerformanceMonitor`
- `MultiTargetMonitor`: 多 target × 单 horizon, 每个 target 独立 `PerformanceMonitor` (不同 baseline / 概率开关)

## 6. 配置层

两级 dataclass:

```
BaseConfig                           # 框架级通用 (model_name, seed, device, log_dir, ...)
└── HPFConfig                        # 业务级, 组合以下子配置
    ├── data:       HPFDataConfig
    ├── feature:    HPFFeatureConfig
    ├── model:      HPFModelConfig
    └── monitoring: HPFMonitoringConfig
```

运行时需要 `dict` 的地方用 `to_adapter_config()` / `to_dict()` 转换:

```python
cfg = HPFConfig()
adapter = HPFAdapter(cfg.to_adapter_config())   # dict → 运行时
```

好处:
- `HPFConfig()` 完整 IDE 补全 / 默认值 / 类型校验
- 转 dict 后才进框架,框架只认 dict,不依赖任何业务 dataclass
- yaml 可读可 diff 可 PR review

### 配置默认值锚定项目根

`HPFMonitoringConfig` 的 `sqlite_path` / `report_dir` / `log_dir` / `alert_log_file` 都用 `field(default_factory=lambda: _root_path(...))` 基于项目根构造绝对路径,**不依赖 CWD**。无论从命令行还是 IDE 直接 Run 文件,日志都落到 `<root>/logs/` 下。

## 7. 导入与打包策略

### src-layout 的理由

```
ROOT/
├── src/tsf_frame/              # 发布包
├── configs/                    # 也发布 (混合 package_dir)
├── pipelines/                  # 不发布, 可执行脚本
└── tests/                      # 不发布
```

好处:
1. `pytest` / `ipython` 在 ROOT 运行**不会**直接读到 `tsf_frame/` 源目录(那里没有,必须走 `src/tsf_frame/`)。强制走"已安装的包"路径,消除"本地能跑、装完就坏"的经典坑。
2. 运行 `pipelines/*.py` 时用 pathlib 向上找到 ROOT,把 ROOT 和 `ROOT/src` 同时插入 `sys.path`,**不 `pip install` 也能直接跑**(对数据科学场景友好)。

### Pipeline / Example 头部模板

```python
from pathlib import Path as _Path
import sys

_HERE = _Path(__file__).resolve()
_PROJECT_ROOT = _HERE.parent  # 兜底
for _p in (_HERE.parent, *_HERE.parents):
    if (_p / 'configs').is_dir() and (_p / 'src').is_dir():
        _PROJECT_ROOT = _p
        for _q in (_p, _p / 'src'):
            if str(_q) not in sys.path:
                sys.path.insert(0, str(_q))
        break

# 写文件路径都基于 _PROJECT_ROOT, 不用 './' 或裸 'logs/'
log_dir = _PROJECT_ROOT / 'logs' / 'runs'
```

### `setup.py` 的混合 `package_dir`

```python
src_packages    = find_packages(where='src')        # → tsf_frame, tsf_frame.business, ...
config_packages = find_packages(where='.', include=['configs', 'configs.*'])

setup(
    package_dir={'': 'src', 'configs': 'configs'},
    packages=src_packages + config_packages,
    ...
)
```

让两个分别住在 `src/` 和 `./configs/` 的顶层包在 `pip install -e .` 后都可导入。

`pip install -e .` 后,实际是通过 `.pth` 指针文件把 `<root>/src/` 注入 `sys.path` (详见 site-packages 下的 `__editable__.tsf_frame-*.pth` + `__editable___tsf_frame_*_finder.py`)。

## 8. 测试策略

`tests/conftest.py` 往 `sys.path` 注入 ROOT + `ROOT/src`,所以:

```bash
pytest tests/                # 不需要先 pip install -e .
```

当前 54 个测试,分布:

| 文件 | 目的 |
|------|------|
| `test_metrics.py` | `MetricsCalculator.calculate_all` 契约 (完美预测 → 误差 0 / R²=1) |
| `test_data_leakage.py` | 特征工程 fit→transform 不窥视未来 |
| `test_adapters.py` | `HPFAdapter` 能被 `HPFConfig.to_adapter_config()` 驱动 |
| `test_monitoring.py` (29 case) | 性能监控 target_ts 对齐、异步真值、各 store 一致性、规则触发、多 horizon、多 target 等 |
| `test_visualization.py` (18 case) | PredictionPlotter 7 原子方法 + 4 复合工具的契约 |

### 关键不变式

监控模块特别注意以下契约:
- `PerformanceMonitor` 用 `target_ts` 对齐, 不依赖位置索引(防回填错位)
- `metric_window <= window_size`, 超过自动 clamp + warning
- `settle_actual` 对已淘汰 target_ts 返回 `False`(明确拒绝)
- 多输出残差按列分位数(不混算)
- 任意 `MetricStore` 实现接口一致: insert/query roundtrip 数据无损

## 10. 扩展新模型

### 10.1 经典 ML 模型

`src/tsf_frame/models/classical/ml_models.py` 加新类 + 挂到 `MODEL_REGISTRY`:

```python
class MyModel(BaseMLModel):
    def __init__(self, config):
        super().__init__(config)
        self.model_name = 'my_model'

    def _build_model(self) -> Any:
        return SomeSklearnEstimator(
            param1=self.config.get('param1', default1),
            ...
        )

# 文件末尾
MODEL_REGISTRY['my_model'] = MyModel
```

`BaseMLModel` 已经处理了:fit 包装多输出 / 残差缓存 / 概率预测路径 / save/load pickle。子类只需实现 `_build_model()`。

### 10.2 深度学习模型

`src/tsf_frame/models/transformer/transformer_models.py`,继承 `_DLBaseModel`:

```python
class MyDLModel(_DLBaseModel):
    def __init__(self, config):
        super().__init__(config)
        self.model_name = 'my_dl'
        # 各种 nn.Module
        self.layer1 = nn.Linear(...)
        ...

        self.criterion = nn.MSELoss()
        # 关键: 必须在 optimizer 创建前调 self.to(self.device)
        # 否则 optimizer 绑定 CPU 参数, GPU 推理时炸
        self.to(self.device)
        self.optimizer = torch.optim.Adam(self.parameters(), lr=...)

    def forward(self, x):
        ...

# 文件末尾
DL_MODEL_REGISTRY['my_dl'] = MyDLModel
```

`_DLBaseModel` 已实现 `fit` (调 `_dl_fit`) / `predict` / `_predict_probabilistic` (MC Dropout)。子类只需实现 `__init__` + `forward`。

**坑提醒:**
- 如果模型用 `BatchNorm`, MC Dropout 推理时打开 `train()` 会让 BN 统计漂掉。换成 `GroupNorm` / `LayerNorm`。
- `self.to(self.device)` 必须在 `self.optimizer = ...` 之前,否则 GPU 配置失效(2026-Q2 修复点)。

## 11. 扩展新监控规则 / 通道 / Store

监控模块的所有插件点都用同一套**注册表 + 装饰器**模式。

### 11.1 新加业务规则

```python
from tsf_frame.monitoring import register_rule, RuleViolation, AlertLevel

@register_rule('R_HPF_LOAN_RATIO')
def rule_loan_ratio(*, prediction, context=None, **_):
    """贷存比不应超过 0.7."""
    deposit = (context or {}).get('total_deposit')
    if deposit and deposit > 0:
        ratio = float(prediction[0]) / deposit
        if ratio > 0.7:
            return [RuleViolation(
                rule_id='R_HPF_LOAN_RATIO',
                severity=AlertLevel.WARNING,
                message=f'贷存比 {ratio:.2%} 超过 70%',
            )]
    return []

# 使用
from tsf_frame.monitoring import RuleEngine
eng = RuleEngine(rule_ids=['R_HPF_LOAN_RATIO'])
```

from tsf_frame.monitoring import AlertChannel, register_alert_channel
import requests

@register_alert_channel('dingtalk')
class DingTalkChannel(AlertChannel):
    def __init__(self, webhook: str, min_level: str = 'warning'):

## 12. 代码规范

- **类型注解**: 所有 public 函数签名必须有类型
- **docstring**: 中英双语,说明参数 / 返回 / 副作用
- **导入路径**: 始终用相对(包内)或绝对包路径(跨包),禁止 `import *`
- **命名**: 类 PascalCase / 函数变量 snake_case / 常量 UPPER_CASE
- **避免在库代码 print**: 用 `get_logger(name, log_dir=...)` 获取 logger
- **绝不在库代码调 `matplotlib.pyplot`**: pipelines/examples 可以,库模块应接收 `ax` 或返回 fig
- **路径锚定项目根**: 库代码 / 配置默认值不要用 `'./logs/...'` 或裸 `'logs/...'`,用 `_PROJECT_ROOT / 'logs' / ...`

## 13. 关键技术决策

### 13.1 XGBoost 一阶差分 + cumsum 还原

**问题**: XGBoost(及其它树模型)的预测被训练集目标的范围 bounding,对长期趋势序列会"贴顶/贴底"。

**解决**: 训练前对目标做 `Δy = y - y.shift(1)`,学习增量;推理时 `y_hat[t] = y_hat[t-1] + Δy_hat[t]` 累加还原。

### 13.2 TimesNet MC Dropout 用 GroupNorm

**问题**: TimesNet 原始用 BatchNorm2d。MC Dropout 推理需要 `model.train()` 保留 Dropout 随机性,但 BN 会因此污染 running stats。

**解决**: 把 `nn.BatchNorm2d` 替换成 `nn.GroupNorm`。GroupNorm 不依赖 batch 统计,`train()` / `eval()` 行为一致。

### 13.3 配置 dataclass → dict 边界

Adapter / Model 只认 dict,不认 `HPFConfig`。这样:
- 从 yaml / json / argparse 进来的任何 dict-like 都能用
- 框架不绑死某个业务的 dataclass

**约定:** dataclass 提供 `to_adapter_config()` / `to_dict()`; 跨业务/跨模块传参**一律用 dict**。

### 13.4 监控:组合而不是继承

通用 `ModelMonitor` 通过**依赖注入**(构造参数)持有各种子系统(store / drift / rule_engine / ...),而不是继承。HPF 业务规则通过 `@register_rule('R_HPF_*')` 装饰器**外挂**注册,不污染通用 monitoring 包。原因:

- 各子系统互相独立,可单测
- 业务定制不需要改框架代码
- 多业务并存(HPF 规则 + Stock 规则)互不干扰

### 13.5 `target_ts` 对齐 + 异步真值回填

HPF 月度跑批场景下,预测和真值不同时到达。`PerformanceMonitor` 用 `target_ts` 作为 `OrderedDict` 的键,而不是 `deque + 位置索引`。即使窗口满 + 真值乱序到达,**回填永远精确**,已淘汰记录会明确返回 `False`。

### 13.6 `metric_window` 与 `window_size` 解耦

- `window_size`: 队列存储容量(保留多少历史 target_ts)
- `metric_window`: 计算指标只看最近 N 条 settled 记录

HPF 推荐 `window_size=36, metric_window=12`(保留 3 年,算近 1 年)。`metric_window > window_size` 自动 clamp 并 warning。

### 13.7 多输出残差按列分位数

多输出模型(同时预测多目标)的残差**按列独立计算**分位数,不被量纲小的维度拖累。`_fit_residuals` 不再 `.flatten()`,`_get_residual_interval` 用 `axis=0` 取分位数。

### 13.8 项目根锚定(防 CWD 飞地)

所有 example/pipeline 顶部必须有 `_PROJECT_ROOT` 解析逻辑,日志/输出路径都基于它构造。配置默认值用 `field(default_factory=lambda: _root_path(...))`。**禁止裸 `'./logs/...'`**(会被 IDE 直接 Run 文件时的 CWD 解析到 `pipelines/examples/logs` 这种飞地)。

## 15. 设计 FAQ

**Q1: 为什么 DL 模型不拆成独立子包?**
A: 保持 `tsf_frame.models` 下 `classical/transformer/moirai` 三类的对称性; 调用侧统一走 `get_ml_model(name, cfg)` / `get_dl_model(name, cfg)`。

**Q2: 为什么没有 `orchestration/` 或 `workflow/`?**
A: `pipelines/*.py` 就是 orchestration。复杂场景上 Airflow/Prefect 时再抽象。

**Q3: 监控为什么默认 SQLite 而不是 Postgres?**
A: 单机、零依赖、能重启回放。`MetricStore` 接口清晰,未来切 Postgres 只需 `@register_store('postgres')` 实现一个 `PgStore(MetricStore)`,业务代码无需改。

**Q4: HPF 业务规则放在 Adapter 还是 Monitor?**
A: 两层都有,职责不同:
- `HPFAdapter.validate_data`: **入口校验**(数据 schema、非负、缺失率) — 进框架前的一票否决
- `RuleEngine` + `@register_rule('R_HPF_*')`: **运行时校验**(看预测结果、上下文、残差) — 监控告警

**Q5: `MultiHorizonMonitor` 为什么不内置在 `ModelMonitor`?**
A: 解耦关注点。`ModelMonitor` 是单 target 单 horizon 的协调器, `MultiHorizonMonitor` 是组合多个 PerformanceMonitor 的轻量包装。两者并列,业务可同时使用(短期告警走 ModelMonitor h=1, 长期体检走 MultiHorizonMonitor 分桶)。

**Q6: 为什么所有 ABC + 注册表都集中在 `interfaces.py`?**
A: 打破循环依赖。各子模块都向下依赖 `interfaces.py`(只有数据类 + ABC,无逻辑),不互相依赖。新加实现:继承 ABC → 装饰器注册 → 工厂取实例,**全过程不动 interfaces.py**。

**Q7: `mixed_feature_handler.py` 是干嘛的?**
A: 历史预留的"混合(时序+静态)特征处理器",当前业务未使用,但保留在代码库供未来扩展(如金融场景同时处理时序行情 + 静态行业属性时)。
